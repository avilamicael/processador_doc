# Phase 06.1: Redesign de Templates e Classificação por Sinais (E/OU + Regex) - Research

**Researched:** 2026-06-18
**Domain:** Evolução de código existente — modelo de dados de sinais, motor de classificação booleano (E/OU + texto|regex), segurança de regex (ReDoS), e reescrita de formulário React.
**Confidence:** HIGH (codebase verificado em sessão; padrões e seams lidos diretamente do código)

## Summary

Esta é uma fase de **evolução de código existente**, não greenfield. Nenhum pacote externo novo é necessário; toda a stack já está instalada e travada no `CLAUDE.md` (FastAPI 0.137 + Pydantic 2.13 + SQLAlchemy 2.0 + Alembic; React 19 + Vite + TS + TanStack Query; SQLite WAL). O trabalho concentra-se em quatro frentes acopladas: (1) trocar `Template.signals_json` de uma lista plana de termos para **grupos E/OU de condições texto|regex**; (2) evoluir `classification/matcher.py` de "fração de termos literais" para **avaliação booleana de grupos**, PRESERVANDO o seam `decide()` (matched/ambiguous/quarantine) que a Fase 5 e o pipeline de revisão dependem; (3) endurecer a execução de regex (guardada como string, compilada SOMENTE no matcher com teto de tamanho — espelhando `validation/fields._MAX_REGEX_LEN`); (4) reescrever `TemplatesPage.tsx` conforme o mockup aprovado (linha densa por campo, construtor de grupos de sinais, dicas via tooltip ⓘ, remoção de `doc_type`).

A decisão de dados mais consequente é **JSON estruturado na coluna `signals_json` existente** vs **novas tabelas (signal_group/signal_condition)**. A pesquisa do codebase mostra que `signals_json` já é uma coluna `Text` lida tolerante a JSON inválido em DOIS lugares (`matcher._signals` e `api/templates._loads_signals`), e que migrações que não tocam `documents` não precisam recriar o trigger `trg_documents_updated_at`. Persistir os grupos como JSON estruturado na mesma coluna é o caminho de menor migração, menor risco e zero schema novo — alinhado ao padrão list-of-pairs (03-01) e ao fato de que sinais NÃO são consultados relacionalmente (são lidos inteiros, por template, no matcher). A migração de dados de sinais planos legados é trivial e pode ser **forward-compatible em leitura** (o parser detecta a forma antiga e a envelopa num único grupo).

A semântica de confiança é o ponto sutil do matcher. Hoje `confidence` é uma fração contínua [0..1] que alimenta `decide(threshold)` e a desambiguação por IA. Com grupos booleanos, a avaliação é binária (casou um grupo ou não). A recomendação é **mapear booleano→confiança preservando o contrato de `decide()`**: grupo casado → confiança alta determinística (ex.: 1.0); nenhum grupo casado → 0.0. Isso mantém `decide()`, a rota de quarentena e a desambiguação por IA funcionando sem reescrever o seam — só a função de pontuação muda.

**Primary recommendation:** Persistir grupos de sinais como JSON estruturado na coluna `signals_json` existente (sem tabelas novas, sem migração de schema — só migração de leitura forward-compatible); evoluir `matcher.match_templates` para avaliação booleana de grupos que produz confiança 1.0/0.0, PRESERVANDO `decide()` intacto; compilar regex SOMENTE no matcher com `re.compile` + teto de tamanho de pattern e de input (espelhar `_MAX_REGEX_LEN`); reescrever `TemplatesPage.tsx` 1:1 com o mockup aprovado.

### Pattern 1: Persistência de estrutura aninhada como JSON em coluna Text (list-of-pairs já adotado)
**What:** Os grupos de sinais são serializados com `json.dumps` na coluna `signals_json` (Text), e desserializados com um parser tolerante a forma antiga/inválida.
**When to use:** Quando a estrutura é lida sempre inteira por uma entidade pai e nunca consultada relacionalmente — exatamente o caso dos sinais (lidos por template no matcher).
**Example (forma proposta dos grupos):**
```python
# signals_json estruturado (proposta). Cada grupo = lista de condições (E);
# lista de grupos = OU. Forma list-of-pairs-friendly, strict-safe.
# Source: derivado do mockup templates-redesign-mockup.html (groups[][])
# [VERIFIED: codebase — mockup linha 139]
[
  [  # grupo 1 (todas E)
    {"mode": "texto", "value": "TryLab"},
    {"mode": "regex", "value": "\\d{44}"}
  ],
  [  # OU grupo 2
    {"mode": "texto", "value": "12.345.678/0001-99"}
  ]
]
```

### Pattern 2: Parser forward-compatible (lê forma antiga E nova)
**What:** O `_signals`/parser detecta se `signals_json` é a lista plana antiga (`["termo1","termo2"]`) ou a nova (lista de grupos de objetos) e normaliza para a forma de grupos.
**When to use:** Para evitar uma migração de dados que reescreva todas as linhas; a leitura adapta. Reduz risco (T-04-01: não tocar `documents`; aqui nem `templates` precisa de UPDATE em massa).
**Example:**
```python
# Source: codebase pattern (matcher._signals tolerante) [VERIFIED: codebase]
def _parse_groups(raw: str | None) -> list[list[dict]]:
    try:
        parsed = json.loads(raw or "[]")
    except (ValueError, TypeError):
        return []
    if not isinstance(parsed, list) or not parsed:
        return []
    # Forma ANTIGA: lista de strings → 1 grupo de condições "texto" (E),
    # OU 1 grupo por termo? Decisão semântica: lista plana legada era
    # "fração de termos" (não E/OU). Mapeamento conservador: cada termo
    # vira UM grupo de 1 condição texto (OU) — preserva "qualquer termo casa".
    if all(isinstance(x, str) for x in parsed):
        return [[{"mode": "texto", "value": s}] for s in parsed]
    # Forma NOVA: lista de grupos (listas de dicts).
    return [
        [c for c in g if isinstance(c, dict)]
        for g in parsed if isinstance(g, list)
    ]
```
> **DECISÃO DE PLANEJAMENTO (precisa confirmação):** o mapeamento da lista plana legada para grupos. A semântica antiga era "fração de termos presentes ≥ limiar". A nova é booleana estrita. Mapear "cada termo = um grupo OU de 1 condição texto" preserva "qualquer termo basta" mas perde a noção de fração. Como NÃO há templates em produção com sinais reais nesta instância de dev (verificar com o usuário), o mapeamento legado pode ser cosmético. `[ASSUMED]` — ver Assumptions Log A1.

### Anti-Patterns to Avoid
- **Compilar/executar regex no endpoint `api/templates.py`:** viola T-04-10 explicitamente documentado na docstring do módulo. A regex de SINAL segue a mesma regra da regex de VALIDAÇÃO de campo: string no endpoint, compilação só no matcher.
- **Reescrever `decide()`:** o seam matched/ambiguous/quarantine é consumido por `classify_stage` (rota de quarentena via `transition(QUARENTENA)`), pela Fase 5 (EM_REVISAO) e pela desambiguação por IA. Mudar sua assinatura ou semântica quebra essas integrações. Só a função de PONTUAÇÃO muda.
- **Criar tabelas signal_group/signal_condition "por simetria" com Template→TemplateField:** simetria estética não é razão; sinais não têm consulta relacional. Custo > benefício.
- **Tocar `documents` na migração:** se houver migração (provavelmente NÃO precisa), ela NÃO pode fazer `batch_alter_table('documents')` — isso dropa o trigger `trg_documents_updated_at` (0002). Padrão consolidado em 0003/0004/0005/0006/0007/0008.
- **Inserir `dangerouslySetInnerHTML` ou interpretar regex/valores como HTML na UI:** valores de sinal/regex são texto puro (T-04-12, padrão de toda a UI da Fase 5).

## Common Pitfalls

### Pitfall 1: Quebrar a semântica de confiança consumida por `decide()` e pela IA
**What goes wrong:** Ao trocar fração contínua por booleano, `decide()` (que compara `best.confidence >= threshold` e a margem `_AMBIGUITY_MARGIN` entre os dois melhores) passa a receber só 0.0/1.0; a desambiguação por IA (zona cinzenta) pode nunca/sempre disparar.
**Why it happens:** `decide()` foi desenhado para confiança contínua; booleano colapsa a "zona cinzenta".
**How to avoid:** Mapear grupo casado → 1.0 e nenhum casado → 0.0. Com isso: vários templates casando (1.0 cada) → margem 0 < `_AMBIGUITY_MARGIN` → "ambiguous" (IA desempata, comportamento correto: dois tipos batem, IA decide); um só casa → "matched"; nenhum casa → "quarantine". **Isso preserva `decide()` SEM alterá-lo.** Escrever teste explícito para os três ramos com a nova pontuação.
**Warning signs:** Testes de `test_matcher.py::test_zona_cinzenta_marca_ambiguo` / `test_nenhum_sinal_vai_para_quarentena` falhando após a mudança.

### Pitfall 2: ReDoS / regex catastrófico colado pelo usuário
**What goes wrong:** Um regex colado (D-T3 — usuário cola o próprio) com backtracking exponencial (`(a+)+$`) trava o worker ao rodar contra um `full_text` longo.
**Why it happens:** `re` da stdlib NÃO tem timeout `[CITED: discuss.python.org/t/107766]`; só a lib 3rd-party `regex` tem `timeout=`.
**How to avoid:** Espelhar a mitigação JÁ existente em `validation/fields.py`: (1) teto de tamanho do PATTERN antes de `re.compile` (rejeita patterns absurdamente longos); (2) teto de tamanho do INPUT (`full_text`/valor) antes de aplicar — `_MAX_REGEX_LEN = 4096` é o precedente; o full_text pode exceder isso, então considerar um teto maior dedicado ao matcher OU aplicar a condição regex sobre janelas/chaves, não o documento inteiro; (3) `try/except re.error` ao compilar → condição não casa (falha fechada, V5), nunca propaga 500. O operador é single-tenant (não atacante), então timeout real (`regex` lib) é desnecessário no v1 — documentar a decisão.
**Warning signs:** Worker travado em 100% CPU num documento; teste com pattern catastrófico não termina.

### Pitfall 3: Migração que recria/dropa o trigger `trg_documents_updated_at`
**What goes wrong:** Se uma migração tocar `documents` (batch recreate), o trigger de `updated_at` some.
**Why it happens:** SQLite + `batch_alter_table` recria a tabela inteira.
**How to avoid:** Esta fase provavelmente NÃO precisa de migração de schema (JSON na coluna existente). Se precisar (ex.: marcar `doc_type` deprecado), seguir o padrão 0003/0004: só CRIA/altera tabelas que NÃO são `documents`. Confirmar no teste `test_migrations.py` que o trigger sobrevive.
**Warning signs:** `updated_at` para de auto-atualizar em documentos após upgrade.

### Pitfall 4: Drift de tipo backend↔frontend (build TS quebra)
**What goes wrong:** Schema de sinais muda no backend mas `types.ts` continua `signals: string[]`; build TS quebra ou (pior) compila com `any` e falha em runtime.
**Why it happens:** Tipos TS são mantidos à mão (não há geração automática de OpenAPI ativa neste projeto — `openapi-typescript` está no CLAUDE.md como recomendado mas não em uso).
**How to avoid:** Atualizar `types.ts`, `api.ts` (tipo do body), `TemplatesPage.tsx` e os schemas Pydantic no MESMO plano/wave. Rodar `tsc`/build do frontend como gate.
**Warning signs:** `npm run build` falhando; `Template.signals` tipado errado.

### Pitfall 5: Condição texto vs regex avaliada contra o haystack errado
**What goes wrong:** Hoje `_haystack` junta `keys` dos pares extraídos + `full_text` em lower. Uma condição "contém texto" deve buscar no CONTEÚDO do documento (full_text), não nas chaves de campo. Regex idem.
**Why it happens:** A função `_haystack` atual mistura chaves de campo (nomes de dados) com o texto do documento — adequado para "fração de termos", mas para sinais que são âncoras do CONTEÚDO (DANFE, CNPJ, chave 44 dígitos) o alvo certo é o `full_text`.
**How to avoid:** Decidir explicitamente o alvo de busca dos sinais. O mockup e D-T2 dizem "busca de texto DENTRO do documento" → alvo = `full_text` (e talvez também os valores extraídos, não só as chaves). Documentar e testar com o exemplo TryLab. `[ASSUMED]` — ver Assumptions Log A2.
**Warning signs:** Sinal "DANFE" não casa um documento que claramente contém DANFE no texto.

## Code Examples

### Avaliação booleana de grupos E/OU (proposta para `match_templates`)
```python
# Source: derivado de matcher.py + mockup (groups[][]) [VERIFIED: codebase]
import re

_MAX_SIGNAL_REGEX_LEN = 512      # teto do PATTERN (V5)
_MAX_HAYSTACK_LEN = 200_000      # teto do INPUT antes de aplicar regex (ReDoS)

def _condition_matches(cond: dict, haystack: str) -> bool:
    value = str(cond.get("value", ""))
    if not value:
        return False
    if cond.get("mode") == "regex":
        if len(value) > _MAX_SIGNAL_REGEX_LEN:
            return False  # falha fechada (V5)
        try:
            pattern = re.compile(value, re.IGNORECASE)
        except re.error:
            return False  # regex inválido nunca casa (V5)
        target = haystack[:_MAX_HAYSTACK_LEN]
        return pattern.search(target) is not None
    # modo "texto": substring literal, case-insensitive
    return value.strip().lower() in haystack

def _group_matches(group: list[dict], haystack: str) -> bool:
    # E: todas as condições do grupo precisam casar; grupo vazio NÃO casa.
    return bool(group) and all(_condition_matches(c, haystack) for c in group)

def _template_matches(groups: list[list[dict]], haystack: str) -> bool:
    # OU: qualquer grupo basta; sem grupos NÃO casa (falha fechada).
    return any(_group_matches(g, haystack) for g in groups)

# match_templates: para cada template, confidence = 1.0 se _template_matches
# senão 0.0. decide() permanece INTACTO (vários 1.0 → ambiguous; um → matched).
```
