# Fase 06.1: Relatório de Revisão de Código

**Revisado:** 2026-06-18
**Profundidade:** standard
**Arquivos revisados:** 9
**Status:** issues_found

## Resumo

Revisão adversarial do redesign de templates (sinais E/OU + regex) e do matcher
booleano. O parser forward-compatible está correto e os dois pontos de
desserialização (`matcher._parse_groups` e `templates._loads_signals_groups`)
produzem resultados idênticos em todos os casos de borda testados (`[]`,
`list[str]`, grupos, condições soltas, tipos mistos) — apesar da diferença
cosmética de guarda `parsed and`. O endpoint HTTP de fato NÃO compila nem executa
regex (guardada só como string; `Literal` rejeita modo inválido com 422). O seam
`decide()` está preservado e o `stage.py` o consome corretamente.

A falha central está na **defesa ReDoS do matcher**: os dois tetos implementados
(`_MAX_SIGNAL_REGEX_LEN` no pattern e `_MAX_HAYSTACK_LEN` no input) NÃO neutralizam
backtracking catastrófico. Um pattern de 5 caracteres contra um input de ~60
caracteres — ambos muito abaixo dos tetos — trava o `re.search` indefinidamente.
Como tanto o pattern (colado pelo operador) quanto o `full_text` (texto extraído do
documento) são dados controlados externamente, isto é explorável e bloqueia o
worker. Detalhe abaixo em CR-01.

## Critical Issues

### CR-01: Tetos de ReDoS não neutralizam backtracking catastrófico — worker trava

**File:** `backend/app/classification/matcher.py:47-51,116-123`

**Issue:** A docstring e os comentários afirmam que o teto de input
(`_MAX_HAYSTACK_LEN = 200_000`) "neutraliza patterns catastróficos (ReDoS)". Isso é
falso. O custo de backtracking exponencial é dirigido por um **substring local
curto**, não pelo tamanho total do haystack. O teto de input só limita varreduras
lineares; não impõe limite de tempo ao engine `re` (stdlib, sem timeout).

Reprodução verificada (ambos os tetos respeitados):
- Pattern: `(a+)+$` → 5 caracteres, bem abaixo de `_MAX_SIGNAL_REGEX_LEN=512`.
- Haystack: `"Relatorio cliente " + "a"*40 + "X resto"` → ~60 caracteres, muito abaixo
  de `_MAX_HAYSTACK_LEN=200_000`.
- `pattern.search(haystack[:200_000])` **não termina** (matou em timeout de 20s e 60s).

Vetores reais no threat model: o operador cola o pattern em modo `regex` (aceito como
string pelo endpoint — por design) e o `full_text` vem do documento ingerido. Um
documento contendo uma sequência longa de um mesmo caractere seguida de quebra é
suficiente para travar a classificação daquele bloco — e, dependendo do modelo de
concorrência do worker, degradar o processamento da fila. O teste
`test_redos_nao_trava` passa apenas porque escolheu um input que casa gulosamente
(`"a"*210010 + "!"`, cortado para 200k de só "a", que casa `$` rápido); ele NÃO
exercita o caso patológico (quase-match que falha) e dá falsa confiança.

**Fix:** O teto de input não basta. Aplicar um limite de TEMPO real ao casamento,
por exemplo trocando o `re` stdlib pelo módulo `regex` (com `timeout=`), ou
executando o `.search` num thread/subprocesso matável com deadline, e tratando o
estouro como "não casa" (mantendo a falha-fechada):

```python
import regex  # pip install regex — suporta timeout no próprio match

_REGEX_TIMEOUT_S = 0.25  # deadline por condição (single-tenant; calibrar)

def _condition_matches(cond: dict, haystack: str) -> bool:
    value = str(cond.get("value", ""))
    mode = cond.get("mode", "texto")
    if mode == "regex":
        if not value or len(value) > _MAX_SIGNAL_REGEX_LEN:
            return False
        try:
            pattern = regex.compile(value, regex.IGNORECASE)
            return pattern.search(
                haystack[:_MAX_HAYSTACK_LEN], timeout=_REGEX_TIMEOUT_S
            ) is not None
        except (regex.error, TimeoutError):
            return False  # falha fechada (inclui ReDoS estourado)
    ...
```

Se trocar de dependência não for aceitável, isolar o `.search` num
`concurrent.futures.ThreadPoolExecutor` com `future.result(timeout=...)` (lembrando
que `re` não é interrompível por thread — então o subprocesso é mais robusto).
Ajustar também a docstring (linhas 22-25, 48-51) para parar de afirmar que o teto de
input "neutraliza ReDoS", e fortalecer `test_redos_nao_trava` para usar o caso
quase-match (`"a"*40 + "!"`) com asserção de tempo (`time.monotonic()` < deadline).

## Warnings

### WR-01: `decide()` pode "casar" ou marcar "ambíguo" sem nenhum sinal quando `threshold <= 0`

**File:** `backend/app/classification/matcher.py:181-193`; `backend/app/config.py:142-145`

**Issue:** A confiança agora é estritamente booleana (1.0/0.0). Em `decide()`, a
guarda de quarentena é `matches[0].confidence < threshold`. Com `threshold = 0.0`
(valor aceito sem validação de faixa em `classify_match_threshold`), um template com
confiança 0.0 (NENHUM sinal casou) satisfaz `0.0 < 0.0 == False` e cai em "matched"
(ou "ambiguous" se houver dois 0.0). Ou seja, um operador que zere o limiar passa a
classificar todo documento contra um template arbitrário, sem nenhum sinal — o
oposto da falha-fechada que o resto do módulo persegue. Verificado:
`decide([0.0], 0.0) == "matched"`; `decide([0.0,0.0], 0.0) == "ambiguous"`.

**Fix:** Tornar a quarentena dependente de evidência real, não só do limiar. Como a
confiança é booleana, exigir confiança estritamente positiva para casar:

```python
if not matches or matches[0].confidence <= 0.0 or matches[0].confidence < threshold:
    return MatchDecision(status="quarantine", template_id=None)
```

Adicionalmente, restringir o campo a faixa válida em `config.py`:
`Field(default=0.5, ge=0.0, le=1.0)`.

### WR-02: `_value_not_blank` rejeita value em branco, mas o matcher trata value vazio como "não casa" — divergência de contrato em dados legados/grupos

**File:** `backend/app/api/templates.py:96-101`; `backend/app/classification/matcher.py:126-129`

**Issue:** O endpoint valida que toda `value` de condição é não-branca (422). Porém o
matcher (`_condition_matches`) e o `_serialize` aceitam silenciosamente condições com
`value` vazio vindas de `signals_json` legado ou de qualquer escrita que não passe
pelo endpoint (ex.: seeds `seed_quarentena.py`/`seed_revisao.py` não revisados, ou
migração futura). Em modo `texto`, value vazio "não casa" (linha 127-128) — ok. Mas
em modo `regex`, value vazio também "não casa" (linha 117) — ok. O risco real é de
CONSISTÊNCIA: o `_serialize` devolve `SignalConditionOut(value="")` para o frontend,
e o `groupsToDrafts` o reidrata como condição visível; ao re-salvar, o endpoint passa
a rejeitar (422) algo que o GET acabou de servir, sem o usuário ter tocado. Round-trip
GET→PATCH quebra para templates legados com condição vazia.

**Fix:** No `_loads_signals_groups`/`_serialize`, descartar condições cujo
`value.strip()` seja vazio ao desserializar (alinhar com o que o matcher já ignora),
para que o que o GET serve seja sempre re-submetível:

```python
groups.append([
    c for c in group
    if isinstance(c, dict) and str(c.get("value", "")).strip()
])
```

### WR-03: condição `texto` faz `.strip()` no needle, mas `regex` não — comportamento silenciosamente diferente entre modos

**File:** `backend/app/classification/matcher.py:116-129`

**Issue:** Em modo `texto`, o valor é normalizado com `value.strip().lower()` antes do
`in`. Em modo `regex`, o `value` é usado cru (sem strip). O endpoint já faz
`v.strip()` na entrada (`_value_not_blank`), mas dados legados / escritas fora do
endpoint não têm essa garantia. O resultado é que um espaço acidental no início/fim de
um pattern regex (ex.: `" \d{44}"`) é interpretado como parte do pattern (casa um
espaço literal), enquanto a mesma sujeira em texto é removida. Isso é uma assimetria
não documentada que produz "não casa" silencioso e difícil de diagnosticar
(o matcher não loga — V7).

**Fix:** Decidir e documentar uma política única. Para regex, NÃO fazer strip (espaços
podem ser significativos no pattern), mas então documentar explicitamente que regex é
byte-exata; para texto, manter o strip. Acrescentar comentário na linha 116 deixando a
assimetria intencional explícita, ou normalizar ambos na borda de escrita
(`_dump_signals`) em vez de só em texto.

### WR-04: edição que limpa todos os sinais envia `signals: []` e apaga silenciosamente os sinais existentes

**File:** `frontend/src/pages/TemplatesPage.tsx:206-235`; `backend/app/api/templates.py:323-324`

**Issue:** No `submitForm`, `signals` é sempre computado e sempre incluído no body do
PATCH (`{ name, signals, fields }`). Se o usuário remover todas as condições (ou
deixá-las em branco), `signals` vira `[]`. No backend, `body.signals is not None` é
verdadeiro para `[]`, então `template.signals_json` é sobrescrito com `"[]"` — os
sinais são apagados. Como o matcher trata "sem grupos" como não-casa (falha fechada),
o template passa silenciosamente a NUNCA mais casar automaticamente, sem aviso ao
operador. Diferente do contrato documentado do PATCH ("omitido preserva"), o form
nunca omite `signals`.

**Fix:** No frontend, ao editar, só enviar `signals` no PATCH quando houver de fato
grupos com condições, ou exibir confirmação quando o save vai zerar os sinais de um
template que tinha sinais. Mínimo: avisar o usuário ("Este template ficará sem sinais
e não casará mais automaticamente — continuar?") antes de submeter `signals: []` numa
edição.

## Info

### IN-01: guarda divergente entre os dois parsers "sincronizados"

**File:** `backend/app/api/templates.py:73` vs `backend/app/classification/matcher.py:95`

**Issue:** `_loads_signals_groups` usa `if parsed and all(isinstance(item, str) ...)`
enquanto `_parse_groups` usa `if all(isinstance(item, str) ...)`. Ambos foram
verificados produzindo resultados idênticos em todos os casos de borda (a diferença só
afeta `[]`, que ambos mapeiam para `[]`), então não é bug hoje. Mas as duas funções se
declaram "réplica EXATA" uma da outra (comentário em `templates.py:54`) e divergem no
texto — qualquer evolução futura de uma corre o risco de não ser refletida na outra.

**Fix:** Extrair UMA função compartilhada (ex.: `app/classification/signals.py`)
importada por ambos, eliminando a duplicação e o risco de drift. Alternativamente,
igualar as duas guardas byte-a-byte e adicionar um teste que afirma
`matcher._parse_groups(x) == templates._loads_signals_groups(x)` sobre um corpus de
casos.

### IN-02: parâmetros mortos mantidos na assinatura pública sem deprecação rastreável

**File:** `backend/app/classification/matcher.py:142-159`

**Issue:** `match_templates` mantém `fields_json` e `doc_type_guess` na assinatura
só por compat e os descarta com `del`. O `stage.py:206-211` ainda os passa. É uma
dívida deliberada e documentada, mas sem marca de remoção (TODO/issue) corre o risco de
ficar permanente, mantendo `extraction.doc_type_guess` vivo só para alimentar um
parâmetro ignorado.

**Fix:** Adicionar um marcador rastreável (ex.: `# TODO(fase-XX): remover params mortos
fields_json/doc_type_guess após atualizar stage.py`) para que a limpeza não se perca.

### IN-03: erro de salvar genérico mascara o 409 de nome duplicado

**File:** `frontend/src/pages/TemplatesPage.tsx:224-235`

**Issue:** O `onError` exibe sempre "Não foi possível salvar o template. Confira os
dados e tente novamente." O backend retorna 409 com mensagem específica para nome
duplicado (`templates.py:300-303`). O usuário que tentar criar um template com nome já
existente recebe um erro genérico em vez de "já existe um template com este nome".

**Fix:** Inspecionar o status/erro no `onError` e mapear 409 → mensagem de nome
duplicado, mantendo o genérico para os demais casos.
