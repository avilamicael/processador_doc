</decisions_in_scope>

<implementation_guidance>
- Hashing por streaming (ler em chunks, ex.: 64KB) para não carregar arquivos grandes inteiros em memória (PITFALLS Performance — "Carregar PDF inteiro em memória").
- Escrita atômica/imutável: gravar primeiro num arquivo temporário dentro do CAS e fazer rename atômico para o caminho final (evita blob meio-escrito; rename é atômico no mesmo filesystem — o CAS está sempre no mesmo volume da pasta de dados). Se o caminho final já existe (mesmo hash), não reescrever (idempotência por conteúdo) e descartar o temporário.
- Tratar o blob final como imutável: não há API de update/delete nesta fase (D-08 mantém para sempre).
- Funções da interface (mínimo): `store(src_path) -> hash`, `path_for(hash) -> Path`, `exists(hash) -> bool`, `open_blob(hash)`/`read_bytes(hash)`. Manter o módulo como a fronteira única do CAS (espelha `storage/db.py`).
- Funciona em Windows: usar `pathlib` e `os.replace` para rename atômico portável; criar subpastas com `mkdir(parents=True, exist_ok=True)`.
</implementation_guidance>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| arquivo de entrada (pasta de origem) → CAS | Conteúdo de documento do cliente é copiado para dentro do sistema |
| CAS (blobs) ↔ filesystem do host | Originais imutáveis (dados fiscais) em repouso na pasta de dados |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-10 | Tampering | integridade do blob no CAS | mitigate | Endereçamento por SHA-256: o caminho é derivado do hash do conteúdo; alteração do conteúdo muda o hash. Escrita via temporário + `os.replace` evita blob meio-escrito. Sem API de update/delete (imutável). |
| T-01-11 | Denial of Service | arquivos grandes em memória ao copiar/hashear | mitigate | Hashing e cópia por streaming em chunks de 64KB (PITFALLS Performance) |
| T-01-12 | Tampering | original na pasta de origem | mitigate | Ingestão COPIA (não move); o original nunca é aberto em modo de escrita (D-07); teste assertando byte-igualdade pós-store |
| T-01-13 | Information Disclosure | blobs (dados fiscais) em repouso | accept | v1 confia nas permissões do `%ProgramData%` (single-tenant); herdado de T-01-03. Criptografia em repouso é evolução. |
| T-01-14 | Repudiation | recuperabilidade após automação (undo) | mitigate | D-08: originais mantidos para sempre, recuperáveis por hash — base de segurança para o undo da Fase 6; teste cobre recuperação após operação posterior |
</threat_model>
