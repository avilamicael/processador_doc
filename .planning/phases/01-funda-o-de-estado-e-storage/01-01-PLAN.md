<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| arquivo de config / env → aplicação | A chave OpenAI e caminhos entram via `.env`/ambiente (entrada de configuração sensível) |
| pasta de dados (banco SQLite) ↔ filesystem do host | Dados fiscais em repouso ficam em `%ProgramData%\ProcessadorDocumentos` |
| cliente HTTP → API (/health) | Qualquer processo na máquina/rede pode chamar a API |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | `app/config.py` chave OpenAI | mitigate | Armazenar como `SecretStr`; `repr`/logs nunca expõem o valor; teste assertando ausência da chave em `repr(settings)` e no corpo de `/health` |
| T-01-02 | Information Disclosure | `/health` endpoint | mitigate | Resposta de `/health` contém apenas status/versão; teste verifica ausência da chave |
| T-01-03 | Tampering | banco SQLite em repouso na pasta de dados | accept | v1 confia nas permissões do `%ProgramData%` do Windows (single-tenant, máquina do cliente); criptografia em repouso é evolução. Pasta criada com permissões padrão do SO. |
| T-01-04 | Information Disclosure | `.env` versionado por engano | mitigate | `.gitignore` ignora `.env`; apenas `.env.example` (sem segredos) é versionado |
| T-01-05 | Denial of Service | escrita concorrente SQLite | mitigate | `PRAGMA busy_timeout` evita falha imediata sob contenção; modelo single-writer (worker da Fase 2) já previsto |
| T-01-SC | Tampering | instalação de pacotes via uv/PyPI | mitigate | Versões pinadas + lockfile (`uv lock`); todas as libs desta fase são mainstream verificadas em STACK.md (fastapi, pydantic, sqlalchemy, alembic, uvicorn). Sem pacotes [ASSUMED]/[SUS]/[SLOP]. |
</threat_model>
