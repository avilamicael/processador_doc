Construir as duas utilidades puras de ingestão isoláveis (sem HTTP, sem DB): o **estabilizador** que decide quando um arquivo da pasta monitorada parou de ser escrito (ING-02 / Pitfall 1), e o **separador** que quebra um PDF em blocos de N páginas via pikepdf (ING-05). Construí-los como utilidades puras com testes próprios permite validar a lógica difícil (quiescência, contagem de blocos) antes de costurá-la.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Pasta do cliente → estabilizador | Arquivo sendo escrito é entrada não-confiável; hash/split sobre conteúdo parcial corrompe dedup |
| PDF do cliente → splitter | PDF malformado/malicioso pode travar o parser (DoS) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03 | Tampering | Hash/split sobre arquivo parcialmente escrito quebra dedup | mitigate | wait_stable exige quiescência size/mtime + lock-test antes de qualquer leitura de conteúdo (Pitfall 1); teste de reinício de contagem prova |
| T-02-04 | Denial of Service | PDF malformado trava o splitter | mitigate | pikepdf (qpdf) é robusto a PDFs malformados; split envolto em try/except levanta exceção controlada (o worker do Plano 03 a roteia para retry→FALHA, não derruba o processo) |
| T-02-05 | Denial of Service | PDF com milhares de páginas → explosão de blocos | accept | Limite de páginas/blocos é discricionário; nesta fase split roda via to_thread (Plano 03) sem bloquear o loop; block_count monitorável; limite explícito fica para evolução |
</threat_model>
