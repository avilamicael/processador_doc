## Resumo

A implementação da ação `copy` está correta nos seus eixos fundamentais: o `executor` acumula cópias como saída adicional sem mutar o alvo do move (D-01); `apply_stage` materializa cópias antes do move (D-03); `undo_operation` discrimina `action="copy"` e apaga só a cópia nunca o original (D-06); o confinamento de path traversal reutiliza `resolve_dest_folder` (V4); a validação HTTP exige `dest_folder` para `copy` (V5). Os 399 testes passam e o fluxo básico copy-only e copy+move foi verificado ao vivo.

No entanto, a inspeção profunda revelou **uma falha de integridade de arquivo** (BLOCKER) que não é coberta pelos testes existentes: a combinação `rename + copy` (sem ação `move`) provoca remoção indevida do arquivo original — violação direta do requisito D-01. Há ainda três problemas de qualidade no frontend que afetam a usabilidade da tela de dry-run com documentos multi-saída.

## Problemas Críticos

### CR-01: `rename + copy` sem `move` remove o original indevidamente — violação D-01

**Arquivo:** `backend/app/automation/stage.py:676-683` (apply_stage) e `backend/app/automation/stage.py:447-454` (dry_run)

**Issue:**
A lógica que decide se deve executar um move efetivo (`has_effective_move`) é inferida comparando o plano-alvo com o valor default, em vez de rastrear diretamente se uma ação `move` foi explicitamente executada. A condição de "alvo default" verifica tanto a pasta quanto o nome:

```python
is_default_target = (
    plan.target_folder is not None
    and Path(plan.target_folder).resolve() == base
    and (plan.target_name is None or plan.target_name == default_name)  # PROBLEMA
)
has_effective_move = not (plan.copies and is_default_target)
```

Quando o usuário configura `rename + copy` (sem nenhuma ação `move`), o executor muta `target_name` para o nome renomeado mas mantém `target_folder` no `base_root`. Nesse cenário:

- `target_folder == base_root` → `True`
- `target_name ('1234') != default_name ('entrada.pdf')` → a condição `target_name == default_name` é `False`
- Logo `is_default_target = False`
- Logo `has_effective_move = not (True and False) = True`

Resultado: `apply_stage` executa o move do CAS para o destino renomeado **e remove o original via `remove_original(source)`**, mesmo que o usuário tenha configurado apenas `copy`. Isso viola D-01 ("a cópia NÃO remove o original"). O mesmo bug afeta `dry_run()`, que exibe incorretamente uma linha de move além da cópia, enganando o usuário no preview.

A combinação não tem teste de integração em `test_stage.py` — os testes de stage cobrem apenas `copy-only` puro (sem rename) e `copy + move` explícito.

**Correção:**
O `AutomationPlan` deve carregar um flag explícito indicando se uma ação `move` foi executada, em vez de inferir isso pelos valores-alvo. Alternativa mais cirúrgica: propagar a informação do executor até o stage.

Em `executor.py`, adicionar campo ao `AutomationPlan`:

```python
@dataclass(frozen=True)
class AutomationPlan:
    ...
    copies: tuple[PlannedCopy, ...] = ()
    has_explicit_move: bool = False   # True só quando uma ação move foi executada
```

Em `_apply_actions` (executor.py), definir o flag ao processar o `move`:

```python
elif action.action_type == "move":
    resolved_folder = naming.resolve_dest_folder(...)
    if resolved_folder is None:
        return None, None, True, (), False
    folder = resolved_folder
    explicit_move = True    # <— rastreia a presença de move
```

Ajustar o retorno para `(folder, name, blocked, copies, explicit_move)` e propagar até `AutomationPlan`.

Em `stage.py`, substituir a lógica frágil por:

```python
# Simples e correto: há move efetivo somente se uma ação move foi explicitamente executada.
has_effective_move = plan.has_explicit_move
```

Isso também resolve o dry_run (mesmo local idêntico).

**Severidade:** BLOCKER — perda de arquivo do cliente; viola D-01 e o requisito de integridade de arquivo do projeto.

## Avisos

### WR-01: Chave React duplicada em `<tr>` para documentos multi-saída

**Arquivo:** `frontend/src/pages/DryRunPage.tsx:317`

**Issue:**
```tsx
<tr key={r.document_id} ...>
```

Quando um documento tem `copy + move`, o backend retorna **duas linhas com o mesmo `document_id`**. React usa a `key` para reconciliação do DOM virtual; chaves duplicadas dentro de uma lista causam comportamento indefinido (atualizações perdidas, re-renders incorretos, erros silenciosos no console). O React renderiza ambas mas pode descartar ou confundir atualizações de estado entre elas.

**Correção:**
Compor uma chave única combinando `document_id` e `action_kind`:

```tsx
<tr key={`${r.document_id}-${r.action_kind ?? 'single'}`} ...>
```

### WR-02: `applicableIds` contém o mesmo `document_id` múltiplas vezes — "Aplicar todos" enfileira duplicatas

**Arquivo:** `frontend/src/pages/DryRunPage.tsx:115` e `239`

**Issue:**
```ts
const applicable = rows.filter(isApplicable)
const applicableIds = applicable.map((r) => r.document_id)
// Para copy+move: applicableIds = [7, 7]

// Botão "Aplicar automações":
onClick={() => doApply(applicableIds)}  // envia [7, 7]
```

Para um documento com `copy + move`, `applicable` contém duas linhas (uma por saída) e `applicableIds` fica com o `document_id` duplicado. `doApply` envia `[id, id]` ao backend, que enfileira o documento duas vezes. A idempotência (`AuditLog done` existente) evita dupla materialização, mas a contagem `enqueued` fica inflada e há trabalho desnecessário no worker.

Os contadores `readyCount`, `skippedCount` e `applicable.length` (exibido como "N documento(s) prontos") também ficam incorretos — um doc com copy+move aparece como 2 documentos prontos.

**Correção:**
Deduplicar por `document_id` antes de exibir contagens e antes de enfileirar:

```ts
const applicableDocIds = [...new Set(applicable.map((r) => r.document_id))]
const readyCount = applicableDocIds.filter(
  (id) => !applicable.some((r) => r.document_id === id && r.skipped_identical)
).length
```

E para "Aplicar todos":
```tsx
onClick={() => doApply(applicableDocIds)}
```

### WR-03: `toggleAll` e `allSel` quebram para documentos multi-saída

**Arquivo:** `frontend/src/pages/DryRunPage.tsx:116` e `120`

**Issue:**
```ts
const allSel = applicable.length > 0 && selected.length === applicable.length
// Para 1 doc com copy+move: applicable.length=2, selected=[id].length=1 => allSel=false
// mesmo com o único documento selecionado

const toggleAll = () =>
  setSelected((s) => (s.length === applicableIds.length ? [] : applicableIds.slice()))
// toggleAll coloca [id, id] em selected
```

Com um documento copy+move, `applicable.length` é 2 mas há apenas 1 documento real. O usuário que seleciona manualmente o documento (via `toggleSel(id)`) vê `selected=[id]` com `applicable.length=2`, então `allSel=false` — o "selecionar todos" nunca aparece como marcado mesmo com todos os documentos selecionados. Inversamente, `toggleAll` com `applicableIds=[id,id]` coloca IDs duplicados em `selected`, tornando `toggleSel` imprevisível (o `.filter(x => x !== id)` remove **todas** as ocorrências de uma vez).

**Correção:**
Trabalhar com conjuntos de IDs únicos para toda a lógica de seleção, usando `applicableDocIds` (ver WR-02):

```ts
const allSel = applicableDocIds.length > 0 && applicableDocIds.every(id => selected.includes(id))
const toggleAll = () =>
  setSelected((s) =>
    allSel ? [] : [...new Set([...s, ...applicableDocIds])]
  )
```

### IN-01: Ausência de teste de integração para combinação `rename + copy` (sem move)

**Arquivo:** `backend/tests/automation/test_stage.py`

**Issue:**
Os testes de stage cobrem `copy-only` puro e `copy + move` explícito, mas não `rename + copy` (sem ação move). Essa lacuna permitiu que o bug CR-01 passasse desapercebido pelos testes. O executor tem `test_copy_inherits_renamed_name_when_rename_runs_first` (puro, sem disco), mas o caminho de integração com `apply_stage` para essa combinação não é exercitado.

**Correção:**
Adicionar em `test_stage.py` um teste `test_rename_copy_no_move_keeps_original` que configura uma automação com `rename + copy` (sem `move`), faz spy em `remove_original` e asserta que ele nunca é chamado e que o doc vai a CONCLUIDO.

### IN-02: `dry_run_result` alias exposto como nome de módulo — oportunidade de refatoração

**Arquivo:** `backend/app/automation/stage.py:483-485`

**Issue:**
```python
# Alias do `dry_run` de módulo — o parâmetro `dry_run` de `apply_stage` sombreia o
# nome da função, então o caminho dry_run=True reusa a simulação multi-saída por aqui.
dry_run_result = dry_run
```

O alias resolve o shadowing local, mas expõe um símbolo público (`dry_run_result`) que não faz parte da interface documentada do módulo (`Interface pública: apply_stage, dry_run, reconcile_orphans, ...`). Não é um bug, mas sinaliza que o design poderia ser mais limpo: extrair a lógica do dry-run para uma função interna `_dry_run_impl` chamada tanto por `dry_run` quanto pelo caminho `dry_run=True` de `apply_stage`.

**Correção:**
Renomear para `_dry_run_impl` (prefixo `_` = privado) e chamar de ambos os pontos, eliminando o alias público.
