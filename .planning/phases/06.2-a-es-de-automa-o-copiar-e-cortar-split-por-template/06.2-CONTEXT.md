# Phase 6.2: Ação de Automação Copiar - Context

**Gathered:** 2026-06-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Esta fase adiciona **uma única ação nova** ao modelo Condições→Ações das automações da Fase 6: a ação **Copiar**. Copiar materializa o arquivo do documento no destino configurado (do CAS, com tokens de campo no caminho, como o Mover) **mas NÃO remove o original** — nem da pasta monitorada, nem em lugar nenhum. É a opção "segura" pedida pelo usuário ("assim não precisa se preocupar tanto com mover um documento e perder ele").

**Cobre:** ação Copiar (criar/editar numa automação, executar no apply, dry-run, undo, audit, anti-colisão), reusando a infraestrutura física da Fase 6.

**NÃO inclui (re-escopado nesta discussão):** a ação **Cortar / split por template** foi **adiada para a v2**. A separação de PDF por pasta monitorada (`pages_per_block`) já existe (Fase 2) e continua; o split dirigido por template precisa de mais reflexão (ver `<deferred>`). Mover, Renomear e o split por pasta **não são alterados** — só não podem regredir.
</domain>

<decisions>
## Implementation Decisions

### Semântica da ação Copiar
- **D-01:** Copiar **materializa o arquivo no destino e deixa o original intacto** (não remove). Implementação = `fileops.materialize_to_dest` **sem** chamar `remove_original`. Novo `action_type = "copy"` no `AutomationAction`; o executor (`automation/executor.py`) despacha por esse rótulo (desconhecido = inerte/falha-fechada, padrão já existente).
- **D-02:** Parâmetros da ação Copiar em `params_json`: pasta de destino (`dest_folder`) com suporte a tokens de campo (`{cliente}`, `{data:aaaa-mm}` etc.), igual ao Mover. O **nome** do arquivo copiado vem da composição com Renomear, como hoje (Renomear muta o nome-alvo; Copiar usa esse nome no seu destino).

### Composição com as outras ações
- **D-03:** **Copiar e Mover coexistem** na mesma automação (ex.: copiar para `Arquivo/` E mover para `Processados/`), e **várias ações Copiar** são permitidas (múltiplos destinos). Ordem de execução: as cópias acontecem primeiro; se houver Mover, ele é o último a agir e remove o original (o original é a garantia até todas as saídas estarem verificadas).
- **D-04:** Mover e Renomear continuam idênticos — **zero regressão** é critério de aceite. Copiar é aditivo.

### Destino do original e estado do documento
- **D-05:** Após Copiar (sem Mover), o **original permanece na pasta monitorada** — isso é intencional (o ponto do Copiar é não perder). A **dedup por hash** já impede reprocessar/cobrar o mesmo arquivo de novo. O documento é marcado **CONCLUIDO** normalmente. Sem sinalização especial de UI no v1 (limpeza manual da pasta fica a cargo do usuário).

### Segurança: dry-run, undo, audit, anti-colisão
- **D-06:** **Undo de Copiar = apaga a(s) cópia(s) criada(s)** no(s) destino(s). É seguro porque o original nunca foi tocado e o CAS preserva o conteúdo → apagar a cópia **nunca causa perda**. Se a automação tinha Mover junto, o undo reverte a **execução inteira** (restaura o original movido + apaga as cópias), como o undo por-run já faz hoje.
- **D-07:** **Audit log write-ahead** registra a intenção de **cada** saída de cópia ANTES de agir (uma automação com N cópias gera N registros write-ahead), reusando o mecanismo da Fase 6. **Anti-colisão** no destino da cópia usa o mesmo `resolve_collision` (sufixo / detecção de duplicata por hash) do Mover — Copiar **nunca sobrescreve** silenciosamente. **Dry-run** mostra origem→destino de cada cópia, deixando claro que o original permanece (sem remoção).

### Claude's Discretion
- Forma exata de expor `action_type="copy"` na API/validação (espelhar a validação de `move`: exige `dest_folder`), na UI do construtor (botão/opção "Copiar" ao lado de Mover/Renomear no editor de etapa) e nos tipos TS — seguindo os padrões já estabelecidos na Fase 6/06-12.
- Se Copiar precisa ou não de migração de schema: `action_type` é uma string livre (`String`), então provavelmente **não há mudança de schema** — só novos valores e lógica de executor/fileops. Confirmar no planejamento.
- Detalhes de dry-run/preview (cor/rótulo da linha de cópia: "copiado", 1→N) seguindo o vocabulário visual já usado.
- Ordem precisa de execução quando há múltiplas cópias + move (cópias antes do move; verificação de hash de cada destino antes de remover o original).
</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `backend/app/automation/fileops.py` — **núcleo do Copiar.** Já tem `materialize_to_dest` (escreve do CAS para o destino, D-11), `resolve_collision` (sufixo/duplicata), `hash_file` (verificação de integridade) e `remove_original` (NÃO usado pelo Copiar). Copiar = materializa + verifica hash, **sem** `remove_original`.
- `backend/app/automation/executor.py` — despacho por `action_type` (`if action_type == "rename" ... elif "move" ...`; desconhecido = inerte, falha-fechada V5). Adicionar ramo `elif action_type == "copy"`.
- `backend/app/models/automation.py` — `AutomationAction.action_type: str` (livre) + `params_json`. Hoje `∈ {rename, move}`; passa a aceitar `copy`. Provavelmente sem migração (string livre) — confirmar.
- `backend/app/automation/undo.py` — undo por-doc e por-run (restaura origem, fallback CAS). Estender para apagar cópias criadas (D-06).
- `backend/app/api/automations.py` — CRUD + validação por `action_type` (rename exige name_pattern; move exige dest_folder). Adicionar validação de `copy` (exige dest_folder).
- Frontend `frontend/src/pages/AutomationsPage.tsx` + `DryRunPage.tsx` + `useAutomations.ts` + `types.ts` — editor de etapa/ação e dry-run/aplicar/desfazer (modelo Condições→Ações de 06-12). Adicionar a opção "Copiar".

### Established Patterns
- **Write-ahead + materializa do CAS + verifica hash antes de "feito"** (AUT-06): Copiar segue o mesmo, só pula o `remove_original`.
- **action_type desconhecido é inerte (falha-fechada)** — adicionar `copy` é seguro: sem o ramo, seria inerte; com o ramo, copia.
- **Idempotência/atomicidade do apply_stage** e dry-run sem tocar disco (Fase 6) — Copiar entra nesse fluxo, não cria caminho paralelo.

### Integration Points
- **Executor:** novo ramo `copy` em `_apply_actions` que produz uma saída de cópia (nome composto + dest_folder próprio).
- **apply_stage/fileops:** executar cópias (materialize sem remove) ANTES do eventual move; cada cópia tem seu write-ahead e verificação de hash.
- **Undo:** registrar as cópias criadas para poder apagá-las na reversão (D-06).
- **API + Frontend:** validação e UI da ação `copy` (espelhar `move` + flag "não remove original").
</code_context>

<specifics>
## Specific Ideas

- Motivação do usuário (verbatim): *"podemos colocar uma automação extra para copiar pdf, assim não precisa se preocupar tanto com mover um documento e perder ele."* Copiar é a ação "à prova de perda".
- *"não vamos remover a opção de mover, manteremos todas"* — Copiar é estritamente aditivo; Mover/Renomear intactos.
</specifics>

<deferred>
## Deferred Ideas

- **Ação Cortar / split de PDF por template → v2.** O usuário decidiu (2026-06-18) adiar: *"talvez seria interessante a gente deixar essa parte de separar PDF pra versão 2. Isso no momento não vai ser necessário."* A separação por pasta monitorada (`pages_per_block`, Fase 2) já cobre o caso simples. O split dirigido por template tem uma questão arquitetural aberta — quando Cortar roda, os pedaços (a) **re-entram no pipeline** (cada um extraído/classificado, mais barato se o corte acontecer ANTES da IA, dirigido por sinais sobre o texto nativo) vs. (b) são **só saída de arquivo** (gera N PDFs e encerra) — além de definir **onde a regra de corte mora no template**. Conecta com a Fase 7 (custo/determinístico, também adiada). Ver memória de projeto `automacoes-acoes-copiar-cortar`.
- **Múltiplos destinos com regras diferentes / sinalização de "já copiado" na UI** — não no v1 (D-05).
